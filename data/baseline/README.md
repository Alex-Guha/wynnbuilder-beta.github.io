# Data Baseline

This directory contains the **source-of-truth** data files that are edited by hand. All other files in versioned `data/<ver>/` directories are generated from these.

## File overview

| File | Purpose | Edited by |
|------|---------|-----------|
| `atree_constants.json` | Ability tree definitions | Hand |
| `major_ids_clean.json` | Major ID definitions | Hand |
| `aspects.json` | Aspect definitions | Hand (stubs from `get_aspects.py`) |
| `clean.json` | Processed item data | Generated by `v3_process_items.py` |
| `tomes.json` | Tome data | Generated by `v3_process_items.py` |
| `ingreds_clean.json` | Ingredient data | Generated by `v3_process_items.py` |
| `recipes_clean.json` | Recipe data | Generated by `process_recipes.py` |
| `tome_map.json` | Tome mapping | Generated by `v3_process_items.py` |
| `dps_data.json` | DPS reference data | Generated by `research/plot_dps.py` |
| `compressed/` | Compressed versions of the above | Generated by `compress_json.py` |

## Updating ability trees / major IDs

1. Edit `atree_constants.json` and/or `major_ids_clean.json` in this directory.
2. From `py_script/`, run `python3 atree-generateID.py` to compile and minify.
3. Copy the output files into the target `data/<ver>/` directory (see full suite below).

## Generating the full data suite for a new version

When a new game version drops and you need to create a complete `data/<ver>/` directory (see `data/2.2.0.14/` for an example of the expected output), follow the workflow in `py_script/README.md`. The condensed steps are:

```bash
# 0. Set the version
VER=X.X.X.X

# 1. Create the version directory
mkdir -p data/$VER

# 2. Generate item/tome/ingredient data from API
cd py_script
python3 v3_process_items.py

# 3. (Optional) Update recipes
python3 process_recipes.py ../data/baseline/recipes_clean.json ../data/baseline/recipes.json

# 4. Check for new aspects and update stubs
python3 get_aspects.py

# 5. Manually update aspects, major IDs, and atree in data/baseline/ as needed

# 6. Compile/minify atree, major IDs, aspects
python3 atree-generateID.py

# 7. Generate DPS data
python3 research/plot_dps.py

# 8. Copy all generated files into the version directory
cp atree_constants_min.json ../data/$VER/atree.json
cp aspects_min.json         ../data/$VER/aspects.json
cp major_ids_min.json       ../data/$VER/majid.json
python3 compress_json.py ../data/baseline/clean.json ../data/$VER/items.json
python3 compress_json.py ../data/baseline/clean.json ../data/baseline/compressed/compress.json
python3 compress_json.py ../data/baseline/ingreds_clean.json ../data/$VER/ingreds.json
python3 compress_json.py ../data/baseline/tomes.json ../data/$VER/tomes.json
python3 compress_json.py ../data/baseline/recipes.json ../data/$VER/recipes.json
cp ../data/baseline/dps_data.json ../data/$VER/dps_data.json

# 9. Generate encoding constants (preview first, then write)
python3 encoding_gen_const.py $VER
python3 encoding_gen_const.py $VER --write
```

### Quick one-liner (atree/aspects/majid/items only)

For smaller updates that only touch atree, aspects, major IDs, or items:

```bash
VER=X.X.X.X

cd py_script && python atree-generateID.py && cp atree_constants_min.json ../data/$VER/atree.json && cp aspects_min.json ../data/$VER/aspects.json && cp major_ids_min.json ../data/$VER/majid.json && python compress_json.py ../data/baseline/clean.json ../data/$VER/items.json && compress_json.py ../data/baseline/clean.json ../data/baseline/compressed/compress.json
```

## Bumping DB version constants

When baseline data files change, the corresponding IndexedDB version constants in `js/data/load_*.js` must be incremented so that browsers invalidate their cached data:

| Baseline file changed | Constant to bump | Location |
|----------------------|------------------|----------|
| `clean.json` | `ITEM_DB_VERSION` | `js/data/load_item.js` |
| `ingreds_clean.json` | `ING_DB_VERSION` | `js/data/load_ing.js` |
| `tomes.json` | `TOME_DB_VERSION` | `js/data/load_tome.js` |
| `aspects.json` | `ASPECT_DB_VERSION` | `js/data/load_aspect.js` |

Note: `atree.json` and `majid.json` are loaded directly from the versioned data directory (not via IndexedDB), so they don't have a DB version constant.
